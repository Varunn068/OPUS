Variables & Types
let, const, var — what to use in test files and why
let vs const vs var USE IN TESTS ▾

In Playwright TypeScript tests, always prefer const for locators and page references. Use let for values that change. Never use var.

const page = browser.newPage();         // ✅ const — never reassigned
const loginBtn = page.locator('#login'); // ✅ const — locator ref

let retryCount = 0;                      // ✅ let — value changes
let response;                              // ✅ let — assigned later

var x = 1;  // ❌ avoid — function-scoped, hoisting issues
KeywordScopeRe-assignableUse in tests
constBlockNoLocators, page, configs
letBlockYesCounters, API responses
varFunctionYes❌ Don't use

Arrow Functions
The syntax you'll write in every Playwright callback
Syntax variants — from simple to multiline USED EVERYWHERE ▾ 
// 1. Single expression — implicit return
const double = x => x * 2;
double(5);  // 10

// 2. No args — needs parens
const getUrl = () => 'https://example.com';

// 3. Multiple args
const add = (a, b) => a + b;

// 4. Multiline — needs curly braces + return
const login = async (page, user) => {
  await page.fill('#email', user.email);
  await page.fill('#password', user.password);
  await page.click('button[type=submit]');
};

// 5. Return object — wrap in parens!
const makeUser = (name) => ({ name, role: 'tester' });
Common mistake: Forgetting async before arrow functions in Playwright. Every function that uses await must be async.

this keyword — why arrow functions are safer in tests ▾

Arrow functions inherit this from the surrounding scope (lexical). Regular functions create their own this.

// Problem with regular function in class
class LoginPage {
  constructor() { this.url = '/login'; }
  
  setup() {
    setTimeout(function() {
      console.log(this.url); // ❌ undefined — 'this' is lost
    }, 1000);
  }
}

// Fixed with arrow function
class LoginPage {
  constructor() { this.url = '/login'; }
  
  setup() {
    setTimeout(() => {
      console.log(this.url); // ✅ '/login' — arrow inherits 'this'
    }, 1000);
  }
}
Note: Don't use arrow functions as class methods if you need to override them in a subclass. Use regular methods for POM classes, arrow functions for callbacks.

Async / Await
The most important JS concept for Playwright — every test uses it
How async/await works CRITICAL ▾

An async function always returns a Promise. await pauses the function until that Promise resolves — without blocking the whole program.

// async function signature
async function getPageTitle(page) {
  const title = await page.title();  // waits for browser response
  return title;
}

// Equivalent arrow function form (used more in Playwright)
const getPageTitle = async (page) => {
  const title = await page.title();
  return title;
};
Rule: Any function that contains await MUST be declared with async. Playwright will throw if you forget.
try/catch for error handling in tests PLAYWRIGHT ▾ 
// Without error handling — test crashes on any error
test('submit form', async ({ page }) => {
  await page.click('#submit');
  await expect(page).toHaveURL('/dashboard');
});

// With try/catch — controlled failure with context
test('submit form', async ({ page }) => {
  try {
    await page.click('#submit');
    await expect(page).toHaveURL('/dashboard');
  } catch (err) {
    console.error('Submit failed:', err.message);
    await page.screenshot({ path: 'submit-fail.png' });
    throw err; // re-throw so test still fails in report
  }
});
Sequential vs Parallel execution PERFORMANCE ▾ 
// Sequential — each awaits the previous (slow)
const user1 = await getUser(1);
const user2 = await getUser(2);
const user3 = await getUser(3);
// Total: user1_time + user2_time + user3_time

// Parallel — all start together (fast)
const [user1, user2, user3] = await Promise.all([
  getUser(1),
  getUser(2),
  getUser(3),
]);
// Total: max(user1_time, user2_time, user3_time)

// Playwright — navigate + intercept in parallel
await Promise.all([
  page.waitForResponse('/api/users'),
  page.click('#load-users'),
]);
Playwright pattern: Use Promise.all([waitForResponse, click]) to avoid race conditions — start listening BEFORE triggering the action.
Common async mistakes in Playwright ▾ 
// ❌ MISTAKE 1: Missing await — test passes before action completes
page.click('#submit');    // no await!
await expect(page).toHaveURL('/done'); // asserts too early

// ✅ FIXED:
await page.click('#submit');
await expect(page).toHaveURL('/done');

// ❌ MISTAKE 2: await inside forEach (doesn't wait)
items.forEach(async (item) => {
  await page.click(item.selector); // forEach won't wait for each
});

// ✅ FIXED: use for...of loop
for (const item of items) {
  await page.click(item.selector); // waits for each click
}
Top bug: forEach does not await async callbacks. Always use for...of when you need sequential awaits in a loop.

Promises
The foundation of async/await — understand this to debug Playwright issues
Promise states & basics ▾

Every Playwright method returns a Promise. Knowing the states helps you debug flaky tests.

StateMeaningIn Playwright
pendingNot yet doneAction in flight (click, navigate)
fulfilledCompleted with valueResponse received, element found
rejectedFailed with errorTimeout, element not found
// Raw promise (you won't often write this, but good to understand)
const p = new Promise((resolve, reject) => {
  if (success) resolve(value);
  else reject(new Error('something failed'));
});

// Consuming with .then/.catch (older style)
p.then(val => console.log(val))
 .catch(err => console.error(err));

// Consuming with async/await (modern — use this)
try {
  const val = await p;
  console.log(val);
} catch (err) {
  console.error(err);
}
Promise.all, Promise.race, Promise.allSettled PLAYWRIGHT ▾ 
// Promise.all — wait for ALL to succeed (fails fast)
const [response, _] = await Promise.all([
  page.waitForResponse('/api/login'),
  page.click('#login-btn'),
]);
console.log(await response.json());

// Promise.allSettled — wait for ALL, even if some fail
const results = await Promise.allSettled([
  checkEndpoint('/api/users'),
  checkEndpoint('/api/orders'),
  checkEndpoint('/api/products'),
]);
results.forEach((r, i) => {
  if (r.status === 'rejected')
    console.log(`Endpoint ${i} failed:`, r.reason);
});

// Promise.race — first one wins (useful for timeouts)
const result = await Promise.race([
  fetchData(),
  new Promise((_, rej) =>
    setTimeout(() => rej(new Error('Timeout')), 5000))
]);

test.describe('Login', () => {
  let loginPage;  // accessible in all tests below

  test.beforeEach(async ({ page }) => {
    loginPage = new LoginPage(page);
    await loginPage.open();
  });

  test('valid login', async () => {
    await loginPage.login('user', 'pass'); // ✅ accessible
  });

Closures — functions that remember their scope ▾

A closure is when a function "remembers" variables from its outer scope even after the outer function has returned.

// Classic closure pattern — factory function
function createCounter(start = 0) {
  let count = start;         // this variable is "closed over"
  return {
    increment: () => ++count,
    reset: () => { count = start; },
    get: () => count,
  };
}
const retries = createCounter();
retries.increment();  // 1
retries.increment();  // 2
retries.get();        // 2

// Practical: environment config builder
function makeConfig(baseUrl) {
  return (path) => `${baseUrl}${path}`;
}
const stagingUrl = makeConfig('https://staging.app.com');
stagingUrl('/login');  // 'https://staging.app.com/login'
stagingUrl('/api');    // 'https://staging.app.com/api'
Test use case: Closures power Playwright's Page Object Model — the page reference is captured in the constructor and "remembered" by all methods.
